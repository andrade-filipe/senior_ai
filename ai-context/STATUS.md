## Checkpoint #1 — Pré-OCR invocation (spec 0010) — aprovado 2026-04-20

Evidência do log `ocr.lookup.hash` (spec 0009 T041) no E2E de 2026-04-20 confirmou o que o ADR-0006 não previa: o `LlmAgent` recebe a imagem como `inline_data` Blob mas o SDK `google-genai` não oferece primitiva que referencie esses bytes dentro de uma function-call; o modelo **fabrica** um base64 arbitrário (184 / 223 bytes — mini PNGs alucinados, hashes mudam por turno). Resposta arquitetural: spec 0010 + ADR-0010 adotam o pattern **CLI-orchestrated pre-step**, em que a CLI chama o OCR-MCP via cliente SSE antes do `runner.run_async` e injeta a lista de exames no prompt como texto. Topologia LlmAgent único (ADR-0006) e PII dupla camada (ADR-0003) preservadas. Novos: env `PREOCR_MCP_TIMEOUT_SECONDS` e `PREOCR_MCP_CONNECT_RETRIES`; código de erro `E_MCP_UNAVAILABLE` (exit 5) em ADR-0008.

**Pesquisa de validação pré-aprovação** (2026-04-20): (1) codelabs oficiais Google ADK *Multimodal Tool Interaction* Parts 1 e 2 confirmam o comportamento de alucinação de base64 e prescrevem *"NEVER ask the user to provide base64 data"*; (2) `mcp.client.sse.sse_client` + `mcp.ClientSession` são API pública estável, verificada no venv instalado (`generated_agent/.venv`); (3) alternativa ADK-idiomática (`before_tool_callback` + `ArtifactService`) considerada e registrada como Alternativa 5 do ADR-0010, rejeitada por introduzir subsistema novo (ArtifactService) e manter step 1 probabilístico. Checkpoint aprovado pelo usuário → spec/plan para `approved`, ADR para `accepted`, tasks para `in_progress`. Onda B (RED tests T010–T015) inicia em seguida.
